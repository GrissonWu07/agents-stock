# 实时量化与历史回放快速指南

这份指南只讲当前主程序里的两条量化验证链路：

- `/live-sim` 实时量化
- `/his-replay` 历史回放

它们共享的是统一股票池中的 `quant_enabled=1` 范围，不再共享结果表，也不再使用“接续到实时模拟账户”的旧模式。

## 1. 启用实时量化股票

先在工作台把股票写入统一股票池，再批量执行 `加入量化`。

当前这一步的真实效果是：

- 将 `stock_universe.quant_enabled` 置为 `1`

之后：

- `/live-sim` 会把这批股票当成“实时量化股票”
- `/his-replay` 会在启动任务时冻结这批股票作为“回放股票范围”

## 2. 实时量化

### 页面定位

`/live-sim` 是独立的模拟账户执行台。

它读取：

- `quant_enabled=1` 的股票范围
- live DB 中自己的账户、持仓、成交、信号

它不读取：

- 历史回放结果表

### 当前行为

1. 默认调度周期是 `10` 分钟。
2. 只在交易时段运行。
3. 运行前会刷新行情技术快照。
4. quote TTL 是 `2` 分钟。
5. local-first 命中时不会重复远程拉。

### 自动执行

自动执行仍是模拟账户自动成交，不是实盘下单。

常见跳过原因：

- 缺少有效最新价
- 建议仓位不足一手
- 当前无可卖持仓

## 3. 历史回放

### 页面定位

`/his-replay` 是回放任务和回放结果页。

它读取：

- 启动时冻结的 `quant_enabled=1` 股票范围
- 历史行情和指标缓存
- `xuanwu_stock_replay.db` 里的任务结果

### 当前动作

当前只支持：

- `开始回溯`
- `取消`
- `删除`

当前不支持：

- `continue`
- `past_to_live`
- “接续到实时模拟账户”

### 数据准备

历史回放准备阶段已经做了这些约束：

1. local-first。
2. partial 只补缺口。
3. 负缓存避免空股票反复远程探测。
4. checkpoint 执行阶段不再远程拉实时行情、基础信息或 AI 分析。

## 4. 两条链路的边界

### 实时量化写入

- `xuanwu_stock.db.strategy_signals`
- `xuanwu_stock.db.sim_positions`
- `xuanwu_stock.db.sim_trades`
- `xuanwu_stock.db.sim_account`

### 历史回放写入

- `xuanwu_stock_replay.db.sim_runs`
- `xuanwu_stock_replay.db.sim_run_signals`
- `xuanwu_stock_replay.db.sim_run_trades`
- `xuanwu_stock_replay.db.sim_run_positions`

历史回放不再写：

- `sim_positions`
- `sim_trades`
- `sim_account`
- `sim_account_snapshots`

## 5. 推荐操作路径

1. 在工作台录入或导入股票。
2. 批量执行 `加入量化`。
3. 进入 `/live-sim` 看实时量化表现。
4. 进入 `/his-replay` 做区间验证。

如果只是想做持仓诊断，不需要先启用 `quant_enabled`；直接登记持仓即可。

## 6. 常见误解

### “量化候选池” 还是不是独立池子？

不是。

当前代码里它只是统一股票池上的 `quant_enabled` 视角，兼容层仍叫 `CandidatePoolService`，但底层已经不是独立物理表。

### 历史回放是不是会影响实时模拟账户？

不会。

历史回放只写 replay 库。

### 为什么有时历史回放很快？

因为当前准备阶段已经复用本地历史 K 线和指标缓存，不再在 checkpoint 内重复远程拉数据。
